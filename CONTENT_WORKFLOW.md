# Content Workflow

## 这份文档是给谁看的

这份文档主要给两类人看：

- 老师本人
- 以后接手这个项目的任何 AI

它的目的只有一个：

**说明老师平时怎样交付内容，AI 收到内容后应该怎样处理。**


## 最核心的工作方式

老师平时不是直接写代码，也不是直接写 JSON。

老师最自然的工作方式之一是：

1. 和 AI 互动
2. 产出一份 Markdown 文稿
3. 把这份文稿放进项目
4. 让 AI 继续把它变成网站内容

但不是所有内容都走这条路。

所以这个项目里，老师主要面对的是：

- Markdown 文稿
- 教学内容
- 修改意见

而不是：

- 页面代码
- 数据结构细节
- 前端逻辑


## 内容流转的两层结构

以后默认把内容分成两种主要工作流来理解：

### 工作流 A：有 Markdown 文稿的持续更新型内容

这是老师会长期继续追加、修改、复用的内容。

目前最典型的就是：

- BBC Six Minute English

这类内容通常有一份 Markdown 文稿，里面会有：

- 标题
- 原文 / 听力文本 / 讲义内容
- 题目
- 答案
- 解析
- 补充说明

这类内容的特点是：

- 适合人阅读
- 适合课堂使用
- 适合老师修改


### 工作流 B：直接生成的网站内容

这类内容不一定先有 Markdown 文稿。

目前最典型的是：

- IELTS Reading
- IELTS Listening
- 部分词汇内容

这类内容可能直接来自：

- 真题 PDF
- 原始题本
- 老师临时给 AI 的材料

AI 收到后可以直接整理成网站内容，不要求老师先额外做一份 Markdown 文稿。


### 网站内容层

无论走哪一种工作流，AI 最终都要整理出网站实际使用的内容。

它可能是：

- JSON 数据
- 目录文件
- 页面引用内容

这层内容的特点是：

- 适合网站读取
- 格式稳定
- 适合复用


## 一句话理解

对于 BBC 这类内容，老师写的是“文稿”，AI 负责把文稿翻译成“网站能用的内容”。

对于 IELTS 这类内容，老师给的是“原始材料”，AI 负责直接整理成“网站能用的内容”。


## 老师最常见的几种交付方式

### 1. 新增一份 BBC 或其他持续更新内容

最常见的情况是：

老师会放进来一份新的 Markdown 文稿。

这份文稿里通常已经包含：

- 题目
- 答案
- 解析
- 原文或材料

AI 收到后应理解为：

“请把这份文稿整理成网站可用内容，并挂到正确位置。”


### 2. 新增一份 IELTS Reading / IELTS Listening / 词汇内容

这类内容不一定有独立的 Markdown 文稿。

老师可能会直接给 AI：

- 一份 PDF
- 一份原始试题
- 一份需要整理的材料

AI 收到后应理解为：

“请直接从原始材料生成网站内容，不要求老师额外先写内容文稿。”


### 3. 修改现有内容

课堂上常会出现这种情况：

- 某道题答案错了
- 某段解析要重写
- 某个题目表达不清楚

这时老师通常不会自己去改 JSON。

老师会直接用自然语言告诉 AI：

- 第 3 题答案改成 B
- 第 5 题解析重写
- 把这句话改得更适合学生

AI 收到后默认应做两件事：

1. 修改网站实际使用的内容
2. 如果这份内容有对应的 Markdown 文稿，尽量同步修改


### 4. 生成临时课堂材料

有些内容是临时课堂用的。

例如：

- 临时讲义
- 临时复习页
- 某次课堂活动材料

这类内容允许继续独立存在，不要求一开始就强行纳入主数据结构。

如果它以后反复使用、价值稳定，再考虑转成标准内容。


## AI 收到内容后的默认处理步骤

任何 AI 在接到老师内容后，默认按这个顺序工作：

1. 先判断这份内容属于哪一类。
   例如：BBC、IELTS 阅读、IELTS 听力、词汇、临时讲义。

2. 判断它走哪种工作流。
   如果是 BBC 这类持续更新内容，优先保留 Markdown 文稿作为老师内容层。
   如果是 IELTS 或一次性整理型内容，可以直接从原始材料生成网站内容。

3. 判断它是否应该进入主结构。
   如果是主干内容，尽量整理成标准内容格式。
   如果是临时讲义，可以先独立存在。

4. 如果有 Markdown 文稿，读取老师的 Markdown 文稿。

5. 提取出网站需要的信息。
   例如：
   - 标题
   - 内容编号
   - 音频路径
   - 题目
   - 答案
   - 解析

6. 生成或更新网站内容层。

7. 更新首页目录。
   这个项目已经确认：首页目录要逐步数据化，不应长期手写在首页页面里。
   当前建议做法是更新对应栏目下的内容元信息，再生成首页目录文件。

8. 做基本检查。
   例如：
   - 页面能否打开
   - 内容是否显示
   - 题目是否能作答
   - 入口是否能找到

9. 用简单的话告诉老师：
   - 做了什么
   - 还有什么不确定
   - 老师下一步只需要看什么


## BBC 新内容的推荐审核流程

如果老师只提供 BBC 原文 Markdown，AI 不应直接把题目写进项目。
推荐流程是：

1. 先生成教师审核版，放在仓库外的临时位置。
2. 教师审核版每道题直接显示题目、选项、答案、证据和技能标签。
3. 证据引用原文 1-3 句左右，并只标 Markdown 行号，例如 `L23-L25`。
4. 不使用 `PXX-LXX` 这种段落编号。
5. 不单独写 `Why`；把判断依据放进 `Evidence`。
6. 老师确认后，再生成网站数据和 CloudBase 私有答案数据。

BBC 填空题在网站数据中必须使用 exactly five underscores:

```text
_____
```

不要使用 `________`。当前 `bbc.html` 只会把第一个 `_____` 替换成输入框，
多余下划线会直接显示在输入框后面。

新增 BBC 内容时，网站能打开不代表学生端 Library 会显示，也不代表能评分。
需要同时完成：

- 静态文件：`data/BBC-*.json`、`content/bbc-six-minute-english/BBC-*.json`、
  `bbc-audio/*.mp3`、`data/home-catalog.*`
- CloudBase `sets`：学生端 Library / Explore 依赖这里
- CloudBase `grading_keys`：提交评分依赖这里

如果直接打开练习可以加载，但学生端 Library 不显示，优先检查 `sets` 是否有
对应 `set_id`。如果提交时报 `GRADING_KEY_NOT_FOUND`，优先检查
`grading_keys` 是否有对应 `set_id`。


## 老师不需要做的事

老师不需要：

- 自己写 JSON
- 自己去首页挂链接
- 自己改页面逻辑
- 自己查哪里出了技术问题

这些默认都属于 AI 的工作范围。


## 老师最好提供的内容

为了让 AI 更顺利工作，如果老师交付的是 Markdown 文稿，最好尽量清楚包含：

- 这是什么内容
- 它属于哪个栏目
- 标题是什么
- 题目和答案分别是什么
- 解析写在哪里
- 如果有音频，音频文件名是什么

不要求老师写成技术格式，只要结构清楚即可。


## BBC 内容源的现实情况

目前 BBC Six Minute English 已经有一个实际在使用的 Markdown 内容库。

它位于老师自己的资料文件夹中，文件命名方式比较稳定，例如：

- `250911 What causes hearing loss.md`
- `260319 How Do We Adapt to the Cold.md`

这说明 BBC 内容后续应继续把 Markdown 文稿视为重要内容源，而不是只把网站里的 JSON 当成唯一来源。


## 正式内容进入项目后的推荐落点

当一份内容已经准备好进入网站正式结构后，AI 应尽量把它放到对应栏目下。

例如：

- BBC 正式内容元信息放在 `content/bbc-six-minute-english/`
- IELTS Reading 正式内容元信息放在 `content/ielts-reading/`

也就是说：

- 老师内容源和网站正式内容可以不是同一份文件
- 但网站正式内容也不应继续全都堆在首页里


## 关于首页目录

这个项目已经决定：

**首页目录从一开始就要往数据化方向建设。**

意思是：

以后新增主干内容时，AI 不应长期依赖手动修改首页代码中的列表。

更理想的做法是：

- 内容有自己的目录信息
- 首页读取目录来展示内容

这样以后新增内容时，流程会更稳定，也更适合交给不同 AI 接手。


## 关于首页栏目显示

这个项目也已经决定：

**首页应始终显示全部正式栏目。**

意思是：

- 不因为某个栏目暂时没有内容，就把它隐藏
- 如果栏目还没有正式内容，点开后显示 `Developing`

这样做的原因是：

- 栏目结构更稳定
- 老师和学生都能看到网站的完整方向
- 以后新增内容或合并栏目时更容易调整

目前首页按三条学习路径组织：

- `Basics`：`BBC Listening`、`Vocabulary`、`Grammar`、`Writing`
- `IELTS`：`Reading`、`Listening`、`Writing`、`Speaking`
- `DSE`：`Paper 1 Reading`、`Paper 2 Writing`、`Paper 3 Listening`、`Paper 4 Speaking`

说明：

- `Basics` 放通用英语能力练习
- `IELTS` 和 `DSE` 放考试专项内容
- 暂时没有正式内容的子栏目可以先处于 `Developing` 状态
- 以后子栏目允许调整或合并


## 关于临时讲义

这个项目也已经决定：

**临时讲义继续允许独立存在。**

原因是：

- 它们很多来自不同 AI
- 生成形式不统一
- 目标是课堂先可用

所以这类内容当前不要求强行进入主结构。


## 当前主干内容类型

目前已经确认的主干内容有：

- BBC 听力
- IELTS 阅读
- IELTS 听力
- 词汇

这些内容目前的命名思路也已经基本明确：

- BBC：按节目日期编号，例如 `BBC-250911`
- IELTS Reading / IELTS Listening：按剑雅短编号，例如 `C7-T1-P2`
- Vocabulary：按词库与组别编号，例如 `NGSL-A`、`NAWL-A`

其中：

- BBC 更像持续更新型内容
- IELTS Reading / IELTS Listening 更像一次性整理型内容
- 词汇目前可按实际材料来源灵活处理

AI 后续做结构设计时，应优先服务这几类内容，并接受它们不一定共享同一种内容来源。


## 当前首页目录的推荐维护方式

当前更推荐的维护方式是：

1. 先修改对应栏目下的正式内容元信息
2. 再生成首页目录文件
3. 最后由首页读取目录文件

这比直接手改首页里的列表更稳定，也更方便以后交给不同 AI 接手。


## AI 沟通要求

以后任何 AI 和老师沟通时，请遵守以下方式：

1. 少讲技术术语。
2. 先说“我理解你的需求是什么”。
3. 再说“我准备怎么做”。
4. 如果有结构性不确定，必须先确认。
5. 如果需求明确，可以直接执行。


## AI 不确定时必须确认的事情

如果遇到下面这些情况，AI 不能自己直接拍板：

- 要不要新增一种新的主干内容类型
- 要不要改老师的日常工作方式
- 要不要改变内容文稿的固定结构
- 要不要大改首页组织逻辑
- 要不要把临时材料正式升格为主结构内容


## 这份文档的目标

这份文档不是技术规范。

它的真正目标是：

**不管以后换哪个 AI，只要它先看这份文档，就知道老师是怎么工作的，自己又该怎么接手。**
